# 0.2｜为什么“已经安装了 R 或 Python”仍然跑不动？软件包、环境、依赖与版本

你在网上找到一段单细胞分析代码，作者写着“需要 Python 和 Scanpy”。你确认电脑已经安装 Python，把代码复制进去，却看到：

```text
ModuleNotFoundError: No module named 'scanpy'
```

另一位同学已经用过 Scanpy，运行同一段代码时却出现版本冲突。还有一位同学在 R 中安装 Seurat，安装过程没有报错，但第二天打开另一个项目后，原来的分析反而不能运行了。

这些现象看起来都像“软件坏了”，其实可能来自不同层：

- 电脑找不到 R 或 Python；
- 软件包装进了另一个环境；
- 两个软件包要求互不兼容的依赖版本；
- 代码使用了旧版本中存在、但新版本已经改变的功能；
- 软件包需要的系统组件没有安装；
- 运行代码的 R 或 Python，并不是你以为的那个。

本节不要求你记住所有安装工具。目标是建立一张清楚的“计算环境地图”：看到一段代码或一个报错时，知道应该先检查哪一层，而不是立刻反复重装。

## 先分清五个经常混在一起的词

### 1. R 和 Python：运行代码的语言与解释器

R 和 Python 都是编程语言。你写下的代码需要由相应的软件读取并执行。

初学阶段可以把这个软件理解为“解释器”：R 解释器执行 R 代码，Python 解释器执行 Python 代码。电脑上安装了 Python，只表示它具备执行 Python 代码的基础能力；这不表示 Scanpy、pandas 或其他第三方软件包已经存在。

同一台电脑还可能同时存在多个 Python。例如：

- Windows 自己安装的 Python；
- WSL 中的 Python；
- Conda 环境中的 Python；
- 某个项目虚拟环境中的 Python。

它们的名称都可能显示为 `python`，实际文件位置和可用的软件包却不一定相同。R 也可能因为系统安装、项目库或服务器模块而出现类似情况。

### 2. 软件包：别人已经写好的功能集合

软件包（package）是一组可以重复使用的代码、说明和相关文件。

例如：

- Seurat 是常见的 R 单细胞分析软件包；
- Scanpy 是常见的 Python 单细胞分析软件包；
- ggplot2 为 R 提供绘图功能；
- pandas 为 Python 提供表格数据处理功能。

安装 R 或 Python，好比获得一间基础实验室；安装软件包，好比再把显微镜、离心机和移液器搬进实验室。

这个类比有边界：软件包不是彼此完全独立的仪器。一个软件包经常会调用其他软件包或系统程序，而且它们之间可能有严格的版本要求。

### 3. 依赖：一个软件包还需要哪些组件

如果软件包 A 必须使用软件包 B 才能工作，我们就说 B 是 A 的依赖（dependency）。

依赖还可能继续向下延伸：

```text
分析代码
  → 软件包 A
    → 软件包 B
      → 底层计算库
```

安装工具通常会自动寻找这些依赖，但“自动”不代表一定能够找到兼容组合。

假设软件包 A 要求某个依赖的版本不低于 2.0，而软件包 C 要求同一个依赖必须低于 2.0。两者放进同一环境时，就可能没有同时满足要求的版本。Python 的 pip 可能把这种情况报告为 `ResolutionImpossible`；R 或 Bioconductor 也可能提示某些软件包太旧、太新，或不适用于当前版本。

### 4. 计算环境：一套彼此配合的运行条件

这里的“环境”不是房间，而是一个项目运行时实际使用的条件集合，通常包括：

- 操作系统和硬件类型；
- R 或 Python 的版本与位置；
- 已安装的软件包及其版本；
- 软件包依赖的底层库；
- 部分路径、语言区域或其他设置。

不同工具对“环境”的管理范围并不完全相同。

- Python 自带的 `venv` 可以为项目建立隔离的 Python 软件包目录。
- Conda 环境可以管理 Python、R 和一部分非 Python 的底层依赖。
- R 的 renv 可以为每个 R 项目建立独立的软件包库，并用 `renv.lock` 记录软件包信息。

这些工具的共同目的，是减少不同项目互相干扰。

### 5. 版本：同一个软件也会随时间变化

软件名称相同，不代表行为永远相同。开发者会修复错误、增加功能，也可能修改默认参数、函数名称、文件格式或依赖要求。

版本号经常写成类似 `1.4.2` 的形式，但不要只凭数字位置猜测兼容性。许多项目采用“主版本、次版本、修订版本”的习惯，具体含义仍应以该软件的官方说明和更新记录为准。

对生物信息学项目而言，需要记录的不只是“用了 Scanpy”或“用了 Seurat”，而是至少说明使用了什么版本、从哪里安装、在什么 R 或 Python 和操作系统中运行。

## 为什么一个项目最好有自己的环境？

假设你同时进行两个项目：

- 项目 A 是两年前的单细胞数据复现，需要较旧版本的软件包；
- 项目 B 是新建的空间转录组分析，需要更新版本的软件包。

如果所有软件都安装在同一个公共环境中，为项目 B 更新软件包时，项目 A 可能突然无法运行。反过来，为了保住项目 A 而拒绝更新，又可能让项目 B 无法安装。

更稳妥的做法是：

```text
项目 A → 环境 A → 项目 A 验证过的软件版本
项目 B → 环境 B → 项目 B 验证过的软件版本
```

环境隔离并不会自动保证分析正确。它只是在计算层面减少“修改一个项目，另一个项目跟着坏掉”的风险。生物学问题、样本设计、参数选择和结果解释仍需要单独验证。

## R 和 Python 常见的环境记录方式

| 项目类型 | 常见隔离方式 | 常见记录文件或信息 | 主要记录什么 |
| --- | --- | --- | --- |
| R 项目 | renv | `renv.lock`、`sessionInfo()` | R、操作系统、软件包及其来源和版本 |
| Python 小型项目 | `venv` | `requirements.txt`、Python 与 pip 版本 | 当前 Python 环境中需要的软件包 |
| Python 或混合型生物信息学项目 | Conda | `environment.yml` 或其他导出文件 | Python/R、软件包、渠道以及部分底层依赖 |

这些文件不是魔法备份。

- `requirements.txt` 通常记录 Python 软件包要求，但不一定包含操作系统中安装的所有底层程序。
- `environment.yml` 可以帮助别人重建 Conda 环境，但不同操作系统能获得的软件包和构建版本可能不同。
- `renv.lock` 可以记录 R 项目的软件包状态，但仍需要合适的 R、操作系统和编译条件。

因此，“可以重建”比“我记得自己装过什么”可靠得多，但不等于在任何电脑上都必然得到完全相同的环境。

## 先确认：代码究竟由哪个 R 或 Python 运行？

下面的命令不是安装教程，而是定位问题时的检查工具。

### 在终端中检查 Python

```bash
python --version
python -m pip --version
python -c "import sys; print(sys.executable)"
```

它们分别回答：

1. 当前 `python` 是什么版本；
2. 当前 Python 调用的 pip 在哪里；
3. 当前 Python 解释器的实际文件路径是什么。

这里特意写成 `python -m pip`，是为了让 pip 由当前这个 Python 调用。直接输入 `pip` 时，系统有可能找到属于另一个 Python 的 pip。

某些 Linux 系统把命令命名为 `python3`。如果官方教程或服务器说明要求使用 `python3`，应按该环境的实际命名操作，而不是盲目把所有命令替换成 `python`。

### 在终端和 R 会话中检查 R

先在终端运行：

```bash
R --version
```

进入 R 后，再运行：

```r
R.version.string
.libPaths()
sessionInfo()
```

它们分别显示 R 版本、R 当前寻找软件包的目录，以及当前会话中的 R、操作系统和已加载软件包信息。

注意运行位置：`R --version` 是终端命令；`R.version.string`、`.libPaths()` 和 `sessionInfo()` 是进入 R 后执行的 R 代码。把它们放错位置，本身就会产生报错。

### 如果使用 Conda

```bash
conda env list
```

输出中的星号通常表示当前激活的环境。安装软件包之前先确认环境名称，可以避免把软件装进 `base` 或另一个项目环境。

## 五类常见报错分别在提示什么？

| 报错或现象 | 更可能对应的层 | 第一项检查 |
| --- | --- | --- |
| `command not found` 或“不是内部或外部命令” | 系统找不到程序 | 程序是否安装；终端能否找到它的路径 |
| `ModuleNotFoundError` | 当前 Python 中没有相应模块 | 当前 Python 路径；软件包是否装在同一个环境 |
| `there is no package called ...` | 当前 R 软件包库中没有该包 | `.libPaths()`；软件包是否装在当前 R 使用的库中 |
| `ResolutionImpossible` 或依赖冲突 | 软件包版本要求不能同时满足 | 哪两个包对同一依赖提出了冲突要求 |
| `cannot open shared object file`、`DLL load failed` | 底层系统库或编译组件 | 操作系统、处理器类型、缺少的库名和软件官方要求 |

报错表只能告诉你“先看哪里”，不能只凭一行文字确定唯一原因。真正求助时，应保留完整命令和从第一条错误开始的完整输出。

## 遇到安装或运行报错时，按这个顺序排查

### 第一步：不要立刻连续重装

先保存报错原文。反复运行不同安装命令可能改变环境，使原始问题更难复现。

### 第二步：确认命令运行在哪里

记录你使用的是：

- Windows PowerShell；
- WSL/Linux 终端；
- R 控制台；
- Python 或 Jupyter；
- 远程服务器。

“在 Windows 电脑上操作”不能说明命令一定运行在 Windows 中，因为 WSL 和远程服务器都可能提供 Linux 环境。

### 第三步：确认实际解释器和环境

记录 R 或 Python 的版本、实际路径，以及 Conda 或项目环境名称。软件包“安装过”却仍然找不到时，这一步尤其重要。

### 第四步：找到第一条有意义的错误

安装日志可能很长。最后一行有时只是“安装失败”，真正原因可能更早出现，例如缺少某个系统库，或两个依赖版本冲突。

从第一条 `ERROR`、`Error`、`cannot`、`failed` 或明确冲突说明开始阅读，再看它前后的几行。

### 第五步：对照软件的官方安装要求

确认：

- 支持哪些操作系统和处理器架构；
- 支持哪些 R 或 Python 版本；
- 是否要求编译器或系统库；
- 应从 CRAN、Bioconductor、PyPI、Conda 渠道还是软件官方来源安装；
- 教程针对的是不是另一个软件版本。

不要因为搜索结果中的一条旧命令看起来更短，就绕过官方要求。

### 第六步：只改变一个条件并重新测试

如果决定更换软件包版本、创建新环境或补装系统组件，一次只做一类改变，并保存命令和结果。这样才能判断是哪项改变解决了问题。

## 一个单细胞项目可能同时包含多个环境

“一个项目一个环境”是初学者很有用的原则，但真实项目有时会调用多个彼此独立的软件。

例如：

```text
原始 10x 数据
  → Linux 中运行 Cell Ranger
  → 得到表达矩阵
  → R 环境中使用 Seurat，或 Python 环境中使用 Scanpy
  → 生成质量控制、聚类和可视化结果
```

Cell Ranger、Seurat 和 Scanpy 不一定安装在同一个环境中。重要的是记录每一步：

- 在哪里运行；
- 输入和输出是什么；
- 使用了哪个软件版本；
- 使用了哪个参考基因组或注释版本；
- 相应的日志和环境记录保存在哪里。

同一原则也适用于单细胞多组学和空间组学。分析可能同时使用测序平台配套程序、R/Python 软件包、图像处理工具和参考数据。环境记录的目的不是把它们强行塞进一个环境，而是让每一步的计算条件都能够追踪。

## Codex 可以帮你安装，但不能替你定义“正确环境”

向 Codex 只说“Scanpy 装不上”，得到的建议往往只能靠猜。更有效的提问应包括：

```text
我在 WSL Ubuntu 中运行命令。
当前 Python 版本是……
python -m pip --version 的输出是……
当前 Conda 环境是……
我执行的完整命令是……
从第一条错误开始的完整输出是……
这个项目的官方安装说明链接是……
请先判断错误属于路径、环境、版本、依赖还是系统库问题，
解释判断依据，再给出最小修改方案。不要直接升级所有软件包。
```

Codex 可以帮助你读取日志、比较版本要求和整理排查步骤，但仍需要你确认：

- 它引用的说明是否对应当前软件版本；
- 命令是否会修改正在使用的环境；
- 更新软件是否可能影响已经验证过的分析；
- 修复后是否重新运行了最小测试和关键结果检查。

会使用 AI 的重要标志，不是让它连续尝试更多命令，而是能向它提供足够的环境证据，并要求每次修改都有可验证的理由。

## 为每个项目保留一份最小环境记录

开始分析时，至少记录：

1. 操作系统及版本；
2. R 或 Python 的版本和实际路径；
3. 项目环境名称；
4. 主要软件包及版本；
5. 安装来源或渠道；
6. 环境文件，例如 `renv.lock`、`requirements.txt` 或 `environment.yml`；
7. 参考基因组、基因注释或其他参考数据版本；
8. 安装和运行日志。

代码能够运行，只说明当前环境接受了这些指令。只有同时保存环境、输入、参数、软件版本和输出检查，其他人才有机会理解并重建这次分析。
