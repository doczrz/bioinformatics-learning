# 0.3｜可以不手写，但必须看得懂：R 基本语法与代码阅读

假设你把一份细胞质量控制表交给 Codex，并要求它筛选可能保留的细胞。它返回了下面这段 R 代码：

```r
cells <- read.csv("cell_metadata.csv")

keep <- !is.na(cells$n_genes) &
  !is.na(cells$percent_mito) &
  cells$n_genes >= 500 &
  cells$percent_mito < 10

filtered_cells <- cells[keep, ]
average_genes <- mean(filtered_cells$n_genes, na.rm = TRUE)
```

即使你不准备亲手从空白页面写出这段代码，也必须能够回答：

- 它读取了哪个文件？
- `cells`、`keep` 和 `filtered_cells` 分别保存什么？
- 哪些细胞会被保留？
- 缺失值怎样处理？
- 最后一行计算的是什么？
- `500` 和 `10` 从哪里来，是否适合你的数据？

如果这些问题无法回答，就不能判断代码是否在处理正确的数据，也不能判断它是否偷偷删除了不该删除的细胞。

本节的目标不是让你背诵 R 语法，更不是要求你脱离文档默写程序。你需要建立的是一种代码阅读能力：看到一段 R 代码时，能够识别数据从哪里进入、经过哪些变化、输出到哪里，以及哪些计算决定需要由研究者负责。

本节假定你已经知道 R 是统计计算与数据分析环境，也知道软件包和项目环境会影响代码能否运行。你不需要提前安装任何软件，也不需要有编程经验。

本节代码块主要用于阅读，无须现在运行。如果希望跟着尝试，应把 R 代码放在 R 控制台或 R 脚本中运行。像 `"cell_metadata.csv"` 这样不带完整路径的文件名，会相对于当前工作目录查找；运行前可以用 `getwd()` 查看这个位置。

第一次阅读时，可以把内容分成两层：

- **必须读懂：**对象、赋值、函数、数据框、索引、逻辑条件、`NA` 和筛选；
- **目前先能辨认：**向量循环使用、稀疏矩阵、`[[]]`、不同管道、命名空间、循环和自定义函数。遇到这些结构时，知道应该要求 AI 解释或查官方文档即可。

## “Vibe coding” 不等于不再需要理解代码

“Vibe coding”是一个非正式说法，通常指使用自然语言描述需求，让 AI 生成、修改并运行代码，而使用者不必逐字输入每条指令。

这种工作方式确实降低了语法记忆的门槛。你可以让 Codex：

- 根据任务生成 R 代码；
- 解释报错；
- 把重复步骤整理成脚本；
- 查找对象在何处被创建或修改；
- 对照文档检查函数参数；
- 运行代码并展示中间结果。

因此，“会不会从头手写”已经不再是衡量分析能力的唯一标准。

但自然语言需求最终仍会被转换为精确的计算操作。计算机不会理解“把质量不好的细胞去掉”这种模糊意图，它只能执行类似“删除 `n_genes < 500` 的行”这样的明确条件。如果 AI 误解了列名、数据单位、缺失值或比较方向，代码仍可能顺利运行。

所以，本课程对初学者的要求是：

> 可以不靠记忆手写代码，但必须能把关键代码翻译成普通语言，并检查这种翻译是否符合研究问题。

这不表示你必须理解 R 的每个内部机制。就像阅读英文论文不要求先成为语言学家，阅读分析代码也可以先掌握最常见的结构，再在需要时查文档。

## 先认识一行 R 代码的组成

看下面这行：

```r
average_genes <- mean(cells$n_genes, na.rm = TRUE)
```

它可以从右向左阅读：

1. `cells$n_genes`：从 `cells` 中取出名为 `n_genes` 的一列；
2. `mean(...)`：调用求平均值的函数；
3. `na.rm = TRUE`：计算前移除缺失值；
4. `<-`：把右侧产生的结果交给左侧名称；
5. `average_genes`：保存结果的对象名称。

把它翻译成普通语言就是：

> 从细胞表中取出检测到的基因数这一列，忽略缺失值后计算平均值，并把结果保存为 `average_genes`。

这类翻译是读懂代码的核心。函数名可以查，参数可以问 Codex，但输入、动作、条件和结果必须能说清楚。

## 对象：R 用名字保存数据和结果

R 创建和处理的内容通常称为对象（object）。数字、文字、表格、矩阵、函数和复杂的单细胞数据容器都可以是对象。

```r
gene_name <- "MS4A1"
cell_count <- 1200
analysis_ready <- TRUE
```

这里创建了三个对象：

| 对象 | 保存的值 | 常见含义 |
| --- | --- | --- |
| `gene_name` | `"MS4A1"` | 文字 |
| `cell_count` | `1200` | 数值 |
| `analysis_ready` | `TRUE` | 逻辑值 |

R 区分大小写，因此 `cell_count`、`Cell_Count` 和 `CELL_COUNT` 是三个不同的名称。

对象名称应该帮助读者理解内容。`cell_metadata` 通常比 `x` 更容易审核，`filtered_cells` 通常比 `result2` 更容易追踪。短名称在公式或很短的临时代码中可能合理，但在长分析流程中会增加误读风险。

## 赋值：`<-` 表示把结果保存到一个名称中

```r
cell_count <- 1200
```

赋值符号 `<-` 可以读作“把右边的值保存到左边”。它由紧挨在一起的 `<` 和 `-` 组成。

R 在许多位置也允许用 `=` 赋值，但常见写法是：

- 用 `<-` 创建或更新对象；
- 在函数调用中用 `=` 指定参数。

```r
average_genes <- mean(cells$n_genes, na.rm = TRUE)
```

这样可以更快区分“保存结果”和“设置函数参数”。

赋值可能覆盖已有对象，而且通常不会再次询问：

```r
threshold <- 500
threshold <- 1000
```

执行第二行后，`threshold` 保存的是 `1000`。阅读 AI 生成的代码时，要特别留意同一个对象是否在后面被重新赋值。

## 注释、引号和括号

### `#` 后面是注释

```r
# 这是解释文字，不会作为计算执行
threshold <- 500  # 暂定阈值，不是通用生物学标准
```

从 `#` 开始直到这一行结束的内容是注释（comment）。好的注释应该解释“为什么这样做”或“这个决定来自哪里”，而不只是把代码重新念一遍。

注释可能过时。代码修改后，旧注释不会自动更新，因此不能只相信注释，还要看实际执行的表达式。

### 引号包住文字

```r
cell_type <- "CD4 T cell"
```

`"CD4 T cell"` 是字符串，也就是文字值。没有引号的 `cell_type` 通常表示对象名称。

```r
cell_type == "CD4 T cell"
```

这行是在比较对象 `cell_type` 是否等于指定文字。

### 圆括号表示函数调用或控制计算顺序

```r
mean(c(10, 20, 30))
```

外层 `mean(...)` 调用求平均值函数；内层 `c(...)` 先把三个数合成一个向量。

括号也可以明确运算顺序：

```r
1 + 2 * 3
(1 + 2) * 3
```

两行结果不同。看到复杂条件时，不应凭视觉猜测优先级；要求 AI 补充括号通常更容易审核。

## 函数：接收输入，完成动作，返回结果

函数（function）可以理解为一项已经命名的操作。

```r
mean(c(10, 20, 30))
```

这行包含：

- 函数名：`mean`；
- 传给函数的输入：`c(10, 20, 30)`；
- 返回值：`20`。

很多函数带有可选参数：

```r
mean(c(10, NA, 30), na.rm = TRUE)
```

`na.rm = TRUE` 表示先移除缺失值，再计算平均值，结果是 `20`。如果没有这个参数，结果通常是 `NA`。

这里 `na.rm` 是参数名称，`TRUE` 是这次函数调用为该参数提供的值。

命名参数比只按位置传值更容易阅读：

```r
read.csv(file = "cell_metadata.csv", check.names = FALSE)
```

你不需要猜测第二个值代表什么。阅读陌生函数时，可以在 R 中查看帮助：

```r
?mean
args(mean)
```

`?mean` 打开函数说明，`args(mean)` 显示常见参数形式。软件包函数还应对照当前安装版本的官方文档，因为参数和默认行为可能随版本变化。

## 向量：一列同类值

向量（vector）是 R 最基础的数据结构之一，可以把它暂时理解为按顺序排列的一组值。

```r
n_genes <- c(820, 410, 1250, 670)
cell_types <- c("T cell", "B cell", "T cell", "Monocyte")
passed_qc <- c(TRUE, FALSE, TRUE, TRUE)
```

`c()` 的含义是 combine 或 concatenate，也就是把多个值连接成一个向量。

一个原子向量中的值会被统一为同一种基本类型。下面的代码把数字和文字放在同一个向量中：

```r
mixed <- c(1, "2", 3)
```

为了让所有元素类型一致，R 会把这些值转换为文字。之后对 `mixed` 直接求平均值就会失败。AI 生成代码时，如果某一列意外混入文字，后续计算可能出现类型转换问题。

### 向量运算通常逐个元素进行

```r
n_genes >= 500
```

如果 `n_genes` 有四个值，结果也是四个逻辑值：

```text
TRUE FALSE TRUE TRUE
```

这不是在回答整列“是否大于等于 500”，而是在逐个判断每个位置。

当两个向量长度不同，R 某些运算会重复使用较短向量的值，这称为循环使用或 recycling。它有时是有意设计，有时会制造不易察觉的错误。看到长度不同的对象参与同一运算时，应先检查 `length()` 或数据维度，不要默认 R 会按照你的生物学分组自动配对。

## 数据框：按行和列组织不同类型的数据

数据框（data frame）是 R 中常见的表格结构。每一列可以有不同类型，但所有列必须对应相同的行数。

下面建立一份完全虚构的细胞表：

```r
cells <- data.frame(
  cell_id = c("cell_1", "cell_2", "cell_3", "cell_4"),
  sample = c("control", "control", "disease", "disease"),
  n_genes = c(820, 410, 1250, NA),
  percent_mito = c(4.2, 12.8, 6.1, 3.5)
)
```

它可以读成：

| cell_id | sample | n_genes | percent_mito |
| --- | --- | ---: | ---: |
| cell_1 | control | 820 | 4.2 |
| cell_2 | control | 410 | 12.8 |
| cell_3 | disease | 1250 | 6.1 |
| cell_4 | disease | 缺失 | 3.5 |

这里的数值只用于解释语法，不是任何单细胞实验的推荐质量控制阈值。真实阈值需要结合组织、实验方案、测序深度、细胞或细胞核类型、分布形状和下游目标决定。

阅读一个陌生数据框时，常见检查包括：

```r
dim(cells)
names(cells)
head(cells)
str(cells)
```

它们分别帮助确认：

- 有多少行和列；
- 列名是什么；
- 前几行长什么样；
- 每列是什么类型，整个对象是什么结构。

这些检查不会证明数据在生物学上正确，但可以尽早发现空表、列名拼错、数值被读成文字或行列方向与预期不一致。

## 矩阵、列表和专用对象：先知道自己在读什么

除了向量和数据框，生物信息学代码中还常见：

| 结构 | 直观理解 | 常见用途 |
| --- | --- | --- |
| 矩阵（matrix） | 行列组成、所有元素通常为同一基本类型 | 基因表达矩阵、降维坐标 |
| 列表（list） | 可以装入不同类型对象的容器 | 保存多个结果、参数或嵌套对象 |
| 专用对象 | 由软件包定义、带有特定结构和方法的对象 | Seurat、SingleCellExperiment 等分析对象 |

同样写成 `object[1]`，不同对象类型的返回结果可能不同。阅读代码前先用 `class(object)` 和 `str(object)` 确认对象类型，比根据对象名称猜测更可靠。

大型单细胞对象还可能使用稀疏矩阵（sparse matrix）。它只重点记录非零值，以减少大量零值带来的存储开销。不要为了“看起来像普通表格”就让 AI 随意转换为普通密集矩阵；这种转换可能显著增加内存占用。

## 索引：`[]`、`[[]]` 和 `$` 都是在取对象的一部分

索引（indexing）用于选择对象中的部分内容。

### R 的位置通常从 1 开始

```r
genes <- c("MS4A1", "CD3D", "LYZ")
genes[1]
```

结果是：

```text
"MS4A1"
```

R 中第一个位置通常写作 `1`，不是许多其他编程语言中的 `0`。

### 数据框使用“行，列”

```r
cells[1, 3]
```

表示第 1 行、第 3 列。

```r
cells[1, ]
```

表示第 1 行、所有列。逗号后留空的含义是“这一维全部保留”。

```r
cells[, "n_genes"]
```

表示所有行、名为 `n_genes` 的列。

### `$` 按名称取数据框的一列

```r
cells$n_genes
```

结果是 `n_genes` 这一列。`$` 左边是对象，右边是其中的名称。

### `[]` 和 `[[]]` 不总是返回同一种结构

对列表或某些数据框操作来说：

- `x[i]` 通常保留一个子集容器；
- `x[[i]]` 通常取出其中一个元素本身；
- `x$name` 按名称取一个元素或一列。

初学阶段不必记住所有边界情况，但必须知道单层和双层方括号不能随意互换。如果 AI 修改了括号形式，要检查返回对象的 `class()`、`length()` 或 `dim()` 是否仍符合下一步要求。

## 比较和逻辑条件：筛选规则藏在这些符号里

常见比较符号包括：

| 写法 | 含义 |
| --- | --- |
| `x == y` | x 是否等于 y |
| `x != y` | x 是否不等于 y |
| `x > y` | x 是否大于 y |
| `x >= y` | x 是否大于等于 y |
| `x < y` | x 是否小于 y |
| `x <= y` | x 是否小于等于 y |

注意：`==` 是比较是否相等，`<-` 是赋值。它们不是同一件事。

逻辑条件可以组合：

| 写法 | 含义 |
| --- | --- |
| `!x` | 不是 x |
| `x & y` | x 和 y 都满足，逐个元素判断 |
| `x \| y` | x 或 y 至少一个满足，逐个元素判断 |

例如：

```r
cells$n_genes >= 500 & cells$percent_mito < 10
```

可以翻译为：

> 对每个细胞，检测到的基因数至少为 500，并且线粒体比例小于 10。

一个符号方向就能改变被保留的细胞。审核筛选代码时，应该把每个条件完整念成一句话，再与分析计划核对。

你也可能在 `if` 条件中看到 `&&` 和 `||`。它们用于形成一次控制流程决策，而 `&` 和 `|` 常用于逐个比较向量。本节不要求掌握它们的所有求值规则，但不要让 AI 在没有解释结果长度的情况下随意互换。

## 缺失值：`NA` 不是 0，也不是文字 `"NA"`

`NA` 表示当前值不可用或缺失。它不等于 0：

- 0 可以表示测量结果为零；
- `NA` 表示没有可用于该位置的值。

`NA` 也不同于字符串 `"NA"`。后者只是由两个字母组成的文字。

检查缺失值应使用：

```r
is.na(cells$n_genes)
```

不要使用：

```r
cells$n_genes == NA
```

缺失值参与比较时，结果通常仍是 `NA`，因为 R 无法确定比较是对还是错。

这会直接影响筛选：

```r
keep <- cells$n_genes >= 500 &
  cells$percent_mito < 10
```

如果 `n_genes` 或 `percent_mito` 缺失，`keep` 中相应位置也可能是 `NA`。更明确的写法是先说明怎样处理缺失值：

```r
keep <- !is.na(cells$n_genes) &
  !is.na(cells$percent_mito) &
  cells$n_genes >= 500 &
  cells$percent_mito < 10
```

这里明确要求 `n_genes` 和 `percent_mito` 都不能缺失，因此 `keep` 只包含 `TRUE` 或 `FALSE`。

这仍然只是一个计算规则，不自动等于正确的科学决定。缺失可能来自导入失败、列类型错误、上游软件未生成指标，或真实的不可测状态。删除之前要先知道缺失为何出现。

## 用逻辑向量筛选行

继续使用虚构的 `cells` 数据框：

```r
keep <- !is.na(cells$n_genes) &
  !is.na(cells$percent_mito) &
  cells$n_genes >= 500 &
  cells$percent_mito < 10

filtered_cells <- cells[keep, ]
```

第一段生成一个与数据行数相同的逻辑向量。第二段使用这个向量保留 `TRUE` 对应的行。

预期结果是：

| cell_id | sample | n_genes | percent_mito |
| --- | --- | ---: | ---: |
| cell_1 | control | 820 | 4.2 |
| cell_3 | disease | 1250 | 6.1 |

这里得到的只是候选元数据行和相应的细胞 ID，并不表示表达矩阵、其他组学模态、降维结果或空间坐标已经同步筛选。真实项目需要使用核对过的稳定标识符筛选所有关联对象，并检查筛选后的标识符、数量和顺序是否一致；具体软件包操作留到相应分析课程。

阅读这段代码时，可以按下面顺序核对：

1. `keep` 的长度是否等于 `nrow(cells)`；
2. `keep` 中是否仍有 `NA`；
3. 筛选前后各有多少行；
4. 被删除的是哪些行；
5. 阈值是否来自预先定义的规则或数据分布检查。

可用于检查的代码包括：

```r
length(keep)
table(keep, useNA = "ifany")
nrow(cells)
nrow(filtered_cells)
```

输出行数符合预期，只能说明筛选按代码执行了；它不能证明阈值在生物学上合理。

## 管道 `|>`：把上一步结果交给下一步

R 代码中经常出现管道：

```r
cells |>
  head(2)
```

基础 R 的 `|>` 会把左侧结果传给右侧函数，所以上面通常可读作：

```r
head(cells, 2)
```

管道的价值是让连续处理步骤按阅读顺序排列：

```r
cells |>
  subset(!is.na(n_genes)) |>
  subset(n_genes >= 500)
```

可以从上到下读成：

1. 从 `cells` 开始；
2. 保留 `n_genes` 不缺失的行；
3. 再保留 `n_genes` 至少为 500 的行。

管道不会自动记录每个中间对象，也不会证明每一步正确。链条过长时，应让 Codex拆成有名称的中间对象，并在关键步骤后显示行列数。

基础管道 `|>` 从 R 4.1.0 开始提供。如果旧环境无法解析它，应先检查 R 版本，而不是盲目把所有管道符号替换掉。你还可能看到软件包 `magrittr` 提供的 `%>%`；两者常用于相似的阅读流程，但细节并不完全相同。

## 软件包代码：先看函数来自哪里

下面两种写法都很常见：

```r
library(dplyr)
filtered_cells <- filter(cells, n_genes >= 500)
```

```r
filtered_cells <- dplyr::filter(cells, n_genes >= 500)
```

`library(dplyr)` 会加载并附加软件包，使其导出的函数可以直接按名称调用。

`dplyr::filter` 明确表示使用 `dplyr` 软件包中的 `filter`。在审核脚本时，这种写法可以帮助确认函数来源，也能减少不同软件包存在同名函数时的歧义。

还要注意缺失值行为：基础 R 使用含 `NA` 的逻辑向量执行 `cells[keep, ]` 时，可能产生内容为缺失值的未知行；`dplyr::filter()` 只保留条件结果为 `TRUE` 的行。不要依赖这种默认差异替你决定缺失值政策，应先明确哪些指标允许缺失以及如何处理。

看到陌生函数时，先问：

- 它属于基础 R 还是某个软件包？
- 软件包是否在当前环境中安装？
- 当前软件包版本是什么？
- 函数是否会修改输入对象、写文件或只返回新对象？
- 默认参数是否适合当前数据？

不要只根据函数名称猜行为。例如，`filter` 在不同软件包或不同上下文中可能不是同一种操作。

## `if`、循环和自定义函数：先看控制结构

AI 生成的脚本可能包含控制流程。你暂时不必自己编写，但要认出它们。

### `if`：条件满足时才执行

```r
if (nrow(filtered_cells) == 0) {
  warning("筛选后没有剩余细胞")
}
```

普通语言翻译是：

> 如果筛选后的数据框有 0 行，就产生一条警告。

花括号 `{}` 包住条件满足时执行的代码。

### `for`：对一组内容重复操作

```r
marker_genes <- c("MS4A1", "CD3D", "LYZ")

for (gene in marker_genes) {
  print(gene)
}
```

这段代码依次把每个基因名交给对象 `gene`，然后打印它。真实循环中还要检查每次结果保存在哪里、文件名是否会互相覆盖，以及某一次失败后循环是否继续。

### `function`：定义可重复使用的操作

```r
calculate_average <- function(values) {
  mean(values, na.rm = TRUE)
}
```

这段代码定义了一个名为 `calculate_average` 的函数。它接收 `values`，返回忽略缺失值后的平均值。

阅读自定义函数时，至少找出：

- 输入参数；
- 函数内部对输入做了什么；
- 默认参数；
- 返回值；
- 是否读取或写入函数外部的对象与文件。

## 报错、警告和消息不是同一回事

R 运行时可能给出不同类型的反馈：

| 类型 | 常见含义 | 首先检查 |
| --- | --- | --- |
| Error | 当前表达式无法继续完成 | 第一条错误、对象是否存在、函数和参数是否正确 |
| Warning | 代码通常继续运行，但发现可能有问题的情况 | 警告对应哪一步、输出是否仍可使用 |
| Message | 软件提供运行状态或说明 | 是否包含版本、方法切换或重要默认行为 |

“没有 Error”不表示没有 Warning，也不表示分析正确。AI 有时会用隐藏消息或警告的函数让输出看起来更整洁；审核时要确认它没有同时隐藏重要证据。

如果 R 控制台出现 `+` 提示符，常见原因是上一行表达式还没有结束，例如少了右括号、引号或花括号。这里的 `+` 是等待继续输入的提示，不是加法结果。

## 读一段 AI 生成的 R 代码，按固定顺序检查

面对几十行代码，不需要从每个符号开始逐字挣扎。先建立数据流。

### 1. 确认运行位置和环境

记录 R 版本、项目环境和软件包版本。代码中的 `|>`、函数参数或对象结构可能依赖版本。

### 2. 找到所有外部输入

搜索读取文件、数据库或网络资源的代码，例如：

```r
read.csv("cell_metadata.csv")
readRDS("seurat_object.rds")
```

确认路径、文件格式、样本身份和参考版本。

### 3. 列出重要对象

把赋值符号左侧的名称列出来：

```text
cells
keep
filtered_cells
average_genes
```

再为每个对象写一句普通语言说明。

### 4. 沿着对象追踪每次变化

检查对象在哪里被创建、筛选、合并、改名或覆盖。不要因为名称里有 `filtered` 就相信它真的按预期过滤过。

### 5. 把所有筛选和比较条件翻译成句子

特别检查：

- `>` 与 `<` 的方向；
- `&` 与 `|`；
- 是否包含等号；
- 缺失值如何处理；
- 条件针对细胞、样本、基因还是其他单位；
- 阈值及分组标签来自哪里。

### 6. 找到写文件或改变系统状态的操作

确认输出路径、文件名和是否可能覆盖已有结果。安装软件包、删除文件和调用系统命令也属于状态改变。

### 7. 检查关键中间结果

在重要步骤后要求显示：

```r
class(object)
dim(object)
head(object)
summary(object)
```

具体检查函数要根据对象类型选择。对大型单细胞对象直接完整打印可能既慢又难读，应查看摘要、维度和关键元数据。

### 8. 最后审核科学含义

代码语法只能说明计算怎样执行。还要问：

- 每一行代表细胞、细胞核、空间位置还是生物学样本？
- 分组标签是否与样本表一致？
- 独立生物学重复是什么？
- 过滤规则是否可能系统性删除某类细胞？
- 基因标识符、参考基因组和注释版本是否匹配？
- 计算结果能支持什么结论，不能支持什么结论？

## 让 Codex 解释代码时，要求它提供可核对的证据

可以把代码和下面的要求一起交给 Codex：

```text
请先不要修改或运行这段 R 代码。

1. 按执行顺序解释每一段代码；
2. 列出所有输入文件、输出文件和被创建的对象；
3. 为每个对象说明类型、预期行列数和生物学含义；
4. 把每个筛选条件翻译成普通中文；
5. 标出缺失值处理、对象覆盖、写文件、删除文件、
   安装软件包和系统命令；
6. 标出依赖软件包版本或默认参数的地方；
7. 区分“语法上能运行”和“科学上已经验证”；
8. 给出最小的只读检查代码，但不要改变原始数据或环境。
```

AI 的解释也需要核对。最直接的核对方式包括：

- 查看当前版本的官方函数文档；
- 用 `class()`、`dim()`、`names()` 和 `str()`检查真实对象；
- 对一个很小的示例运行单步代码；
- 比较筛选前后的行数和被删除对象；
- 保留原始脚本和运行日志。

## AI 生成代码中需要停下来检查的写法

下面这些代码不一定错误，但不能在不理解的情况下直接运行：

| 写法 | 可能影响 | 运行前要问 |
| --- | --- | --- |
| `rm(list = ls())` | 删除当前 R 工作区中的对象 | 是否有尚未保存的对象或证据？ |
| `setwd("某个绝对路径")` | 改变后续相对路径的基准位置 | 这是不是作者个人电脑路径？ |
| `install.packages(...)` | 修改当前 R 软件包环境 | 是否应该在项目环境中安装并记录版本？ |
| `na.omit(data)` | 删除含缺失值的行 | 删除了多少行，缺失为何出现？ |
| `write.csv(..., "原输入文件.csv")` | 可能覆盖输入文件 | 输出是否使用新路径并保留原始数据？ |
| `unlink(...)` | 删除文件或目录 | 目标路径究竟是什么，是否可恢复？ |
| `system(...)` | 调用操作系统命令 | 命令在哪里运行，会读取或修改什么？ |
| `<<-` | 可能修改函数外部的对象 | 为什么需要改变全局状态？ |

还有一些操作看似安全，却可能改变结论：

- 只按列位置而不是列名取数据；
- 未检查单位就比较百分比与小数；
- 直接把每个细胞当成独立生物学重复；
- 在查看分组结果后再调整筛选阈值；
- 把警告全部隐藏；
- 覆盖同名对象，使原始与处理后数据无法区分。

审核的目的不是禁止所有简写，而是知道每项简写省略了什么信息。

## 完整读懂开头的示例

现在重新阅读本节开头的代码：

```r
cells <- read.csv("cell_metadata.csv")

keep <- !is.na(cells$n_genes) &
  !is.na(cells$percent_mito) &
  cells$n_genes >= 500 &
  cells$percent_mito < 10

filtered_cells <- cells[keep, ]
average_genes <- mean(filtered_cells$n_genes, na.rm = TRUE)
```

普通语言解释是：

1. 从当前路径读取 `cell_metadata.csv`，保存为数据框 `cells`；
2. 对每一行检查 `n_genes` 和 `percent_mito` 是否都不缺失、`n_genes` 是否至少为 500，以及 `percent_mito` 是否小于 10；
3. 只有四个条件都满足的行才在 `keep` 中得到 `TRUE`；
4. 使用 `keep` 筛选 `cells`，把结果保存为 `filtered_cells`；
5. 对筛选后细胞的 `n_genes` 计算平均值，并在计算时移除仍然存在的缺失值；
6. 把平均值保存为 `average_genes`。

如果在交互式 R 控制台中单独输入一个未赋值的表达式，它的结果通常会自动显示；通过 `source()` 或其他脚本方式运行时则不应依赖自动显示。需要可靠记录时，应把结果赋值给对象，并根据需要显式运行 `print(average_genes)` 或写入输出文件。

这段代码仍有需要研究者回答的问题：

- 输入文件是否真的是当前项目的细胞元数据？
- 列名和单位是否正确？
- 阈值 `500` 与 `10` 的依据是什么？
- 是否还应检查其他质量指标？
- 每个样本被保留了多少细胞？
- 这些规则是否对某些组织、细胞类型或细胞核数据产生系统性偏差？
- 为什么最后计算的是细胞层面的平均基因数，它要回答什么问题？

能读懂代码，不是能预测每个标点，而是能把计算过程和科学判断分开，并知道还缺少哪些证据。

## 本节需要保留的最低代码素养

面对一段 R 代码，你不必依靠记忆重写它，但至少应该能够：

- 认出对象、赋值、函数、参数、注释和字符串；
- 区分向量、数据框、矩阵、列表和专用对象；
- 看懂常见的索引、比较、逻辑条件和管道；
- 知道 `NA` 需要被明确处理；
- 追踪输入、对象变化和输出；
- 找到可能覆盖对象、修改环境或删除文件的操作；
- 用真实对象结构和官方文档核对 AI 的解释；
- 说明代码执行成功与生物学结论成立之间仍缺少哪些验证。

在 AI 可以快速生成代码的今天，语法记忆的重要性可以下降，但代码理解的重要性反而更高。因为生成速度越快，越需要有人看清它究竟做了什么。
